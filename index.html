Parent Element
When the checkbox inside me is checked, I change — without JavaScript. :has() lets a parent style itself based on its children.

Browser support: Chrome 105+, Safari 15.4+, Firefox 121+. This is arguably the most requested CSS feature in history.

The CSS

/* Parent responds when child is :checked */
.parent:has(input:checked) {
  background: rgba(0,255,136,0.06);
  border-color: rgba(0,255,136,0.3);
  transform: translateX(12px);
}

/* Style a sibling via the parent */
.parent:has(input:checked) .title {
  color: var(--electric);
}

Previously impossible in pure CSS, :has() solves the "parent selector" problem. Use it for form validation states, navigation highlighting, empty state detection, and conditional layouts — all without touching JavaScript.

Each card animates in as it enters the viewport. No Intersection Observer. No JavaScript event listeners.
animation-timeline: scroll()
The progress bar at the top of this box is driven by scroll(nearest block) — it tracks how far you've scrolled.
animation-range
Control exactly when the animation fires — entry 0% entry 30% means "play during the first 30% of entry into view."
Scroll Snapping
Combined with scroll-snap-type and scroll-snap-align, this creates buttery paged scroll containers.

The CSS

/* Card reveal on scroll */
.card {
  animation: card-reveal linear both;
  animation-timeline: view();
  animation-range: entry 0% entry 30%;
}

@keyframes card-reveal {
  from {
    opacity: 0;
    transform: translateY(20px);
  }
}

/* Progress bar */
.progress {
  animation: grow linear;
  animation-timeline: scroll(nearest block);
}

@keyframes grow {
  to { transform: scaleX(1); }
}

Scroll-Driven Animations land in Chrome 115+ and are coming to all browsers. They replace the most common use-case for IntersectionObserver — and they run off the main thread for smooth 60fps performance.

Hover me — I'm styled with nested CSS
  • Nested rules — no preprocessor needed
  • Full browser support as of mid-2024
  • Hover state defined inside the parent rule
  • Replaces Sass/LESS for most use-cases

No build step required. Native CSS nesting means you can write structured, readable CSS without Sass, Less, or PostCSS. Your stylesheet is your source of truth.

The CSS

/* Native nesting — ships in all browsers */
.card {
  background: #111;

  /* Nested child selector */
  & .title {
    color: white;

    /* Nested pseudo-class */
    &:hover { color: var(--electric); }
  }

  & li {
    border-bottom: 1px solid #1e1e1e;

    &:last-child { border-bottom: none; }
    &:hover     { padding-left: 0.5rem; }
  }
}

Container Query
Adaptive Card
This card changes layout based on its container width, not the viewport. Make it wider to see the horizontal layout appear.
WIDE LAYOUT
≥ 400px container

Container queries mean reusable components can be truly adaptive — they respond to where they're placed, not the viewport size. One component, infinite contexts.

The CSS

/* Define the container */
.wrapper {
  container-type: inline-size;
  container-name: card-container;
}

/* Query the container, not the viewport */
@container card-container (min-width: 400px) {
  .card {
    flex-direction: row;
  }

  .card-meta {
    display: block;
  }
}

The holy grail of responsive design — components that know their context. Sidebar widgets, grid cards, and modular UI can all adapt to their container independently of the page layout.

Positioning

Anchor
Positioning

Hover the button — tooltip positions itself

📌 I'm anchored to that button using CSS alone — position-anchor: --my-btn

No getBoundingClientRect(). No popperjs. No JavaScript.

CSS Anchor Positioning replaces an entire category of JS: tooltips, popovers, dropdowns, and floating UI all tethered to their target element without a single line of script. Chrome 125+.

The CSS

/* Name the anchor element */
.btn {
  anchor-name: --my-btn;
  position: relative;
}

/* Position relative to the anchor */
.tooltip {
  position: absolute;
  position-anchor: --my-btn;
  top: calc(anchor(bottom) + 10px);
  left: anchor(left);
}

/* Show on hover */
.btn:hover + .tooltip {
  opacity: 1;
}

Cascade Control

Cascade
Layers

Three layers — last wins

@layer base — colour: #666
@layer components — colour: volt yellow
@layer overrides — colour: coral red (wins)

@layer ends specificity wars. Layers cascade in declared order — later layers win regardless of selector specificity. Third-party CSS can live in a @layer base and never beat your own component styles.

The CSS

/* Declare layer order up-front */
@layer base, components, overrides;

@layer base {
  .el { color: #666; }
}

@layer components {
  /* Wins over base, even lower specificity */
  .el { color: var(--volt); }
}

@layer overrides {
  /* Always wins — it's the last layer */
  .el.override { color: var(--coral); }
}

:is() groups selectors with the highest specificity of its arguments.
:where() does the same but with zero specificity — easy to override.
:not() excludes — hover anything that isn't .disabled.

The CSS

/* :is() — group with highest specificity */
.item:is(.featured, .highlight) {
  color: var(--electric);
}

/* :not() — exclude specific elements */
.item:not(.disabled):hover {
  color: var(--volt);
}

/* :where() — zero specificity, easy override */
:where(.grid) .item.active {
  color: var(--coral);
}

Interactive UI

JS-Free
Accordion

Click any item — native <details> element

What is the <details> element?
The <details> HTML element creates a disclosure widget — no JavaScript needed. Pair it with <summary> for the heading. The open/close animation is handled entirely with CSS transitions on the [open] attribute selector.
Can I style it however I want?
Absolutely. Hide the default marker with ::-webkit-details-marker { display: none } and replace it with any CSS pseudo-element. The [open] attribute selector is your friend for toggling states — rotate icons, swap colours, animate content in.
What about exclusive accordions?
HTML now supports name attribute on <details> to create exclusive groups — only one opens at a time. This is the <details name="group"> pattern, analogous to radio inputs.
Is it accessible?
Yes — <details>/<summary> is natively keyboard-accessible and communicates state to screen readers via ARIA. No additional roles or attributes needed for basic use.

The CSS

/* Hide default arrow */
summary::-webkit-details-marker {
  display: none;
}

/* Custom icon */
summary::after {
  content: '+';
  transition: transform 0.3s;
}

/* Rotate when open */
details[open] summary::after {
  transform: rotate(45deg);
}

/* Animate content in */
details[open] .body {
  animation: details-open 0.3s ease;
}

Tabs

Click the tabs — no JavaScript

The Checkbox Hack
The radio inputs are hidden off-screen. Labels act as clickable triggers. CSS sibling selectors (:checked ~ .panel) show and hide the correct panel. No JavaScript, no event listeners.
The Mechanism
Radio inputs share a name attribute — only one can be checked at a time. The :checked pseudo-class combined with the general sibling combinator (~) reaches the corresponding panel and sets display: block.
Universal Support
This technique works in every browser released in the last decade. The :checked selector and sibling combinators are CSS Level 3 — no flags, no progressive enhancement needed. It just works.

The CSS

/* Hidden radio inputs */
.tab-input { display: none; }

/* Hide all panels by default */
.tab-panel { display: none; }

/* Show active panel via :checked + ~ */
#tab-1:checked ~ .panels .panel-1,
#tab-2:checked ~ .panels .panel-2 {
  display: block;
  animation: fade-in 0.25s ease;
}

/* Active tab indicator */
#tab-1:checked ~ .bar .label-1::after {
  transform: scaleX(1);
}
